# 2026-04-29 v29.6 — procedural chat-style probes + entry-point rotation

**Status:** SHIPPED 2026-04-29 03:20 UTC. Continuation of the
"nothing static" sweep, eliminating the largest remaining static
surfaces left after v29.5 (the chat-style probe pools and the
function-name identifiers in coding-style benches).

## What changed

### 1. Procedural chat-prompt synthesis

A new `_synthetic_chat_prompt` helper generates fresh chat-style
prompts per call from ~12 template families, with slots filled by
a mix of procedural synthesis (topics, openers, characters, quotes)
and small fixed real-word lexicons (audiences, aspects, languages,
styles, tasks — needed for semantic anchoring).

The three chat-probe pickers all switched from "shuffle static
pool + paraphrase" to "synthesise N fresh prompts per round":

| Picker | Before | After |
|--------|--------|-------|
| `_pick_on_policy_rkl_prompts` | shuffle 88-entry `ON_POLICY_RKL_POOL` + `_paraphrase_chat_prompt` | `[_synthetic_chat_prompt(rng) for _ in range(N)]` |
| `_pick_judge_probe_prompts` | shuffle 65-entry `JUDGE_PROBE_POOL` + paraphrase | `[_synthetic_chat_prompt(rng) for _ in range(N)]` |
| `_pick_chat_turns_prompts` | shuffle 25-entry `CHAT_TURNS_PROBE_POOL` + paraphrase | `[_synthetic_chat_turn_chain(rng, n) for _ in range(N)]` |

The legacy pools are kept in source for the `block_seed=None` local-
dev / replay path (deterministic against file content). Production
always supplies a real `block_seed`, so the procedural synthesis
fires every round.

### Template families

Each `_synthetic_chat_prompt` call picks one of:

* `explain` — "Explain {topic} in {N} sentences."
* `describe` — "Describe {topic} for {audience}, in {N} sentences."
* `compare` — "Compare {A} and {B} in terms of {aspect}."
* `continue_story` — "Continue this story in {N}-{N+1} sentences: '{opener}'"
* `define` — "Define '{term}' clearly for {audience}."
* `story` — "Write a 60-word short story about {char} who {action}."
* `list` — "List {N+2} {category} that {qualifier}."
* `opinion` — "What are the practical trade-offs of {choice} versus {alt}?"
* `translate` — "Translate the following sentence into {language}: '{quote}'"
* `rewrite` — "Rewrite the following sentence to be {style}: '{quote}'"
* `howto` — "How would you {task}? Answer in {N} sentences."
* `steps` — "List the steps to {task}. Number each step ..."

Procedural slots: `{topic}`, `{topic_a/b}`, `{opener}`, `{char}`,
`{quote}`, `{term}`. Real-word slots: `{audience}`, `{aspect}`,
`{language}`, `{style}`, `{task}`, `{category}`, `{qualifier}`,
`{choice}`, `{alt}`.

### Multi-turn chains

`_synthetic_chat_turn_chain(rng, n_turns=3)` picks a "scenario"
template (5 chains: dinner-party / blog-post / home-garden /
difficult-conversation / team-retro) and emits N consecutive turns
that naturally chain off each other. The scenario's slots are
filled procedurally (`{topic}` via `_synthetic_topic_phrase`,
`{place}` from a small fixed lexicon, `{char}` via `_synthetic_name`).

### Why the teacher-rubric grading still works

For `judge_probe` and `chat_turns_probe`, the teacher grades the
student's response on a 1-5 rubric (helpfulness / clarity /
correctness / coherence). The rubric measures **how the model
answers**, not what specific facts it cites about the procedural
topic. A coherent response to "Describe the pumkanal weltarstone
for a curious child" is gradeable on rubric quality the same way
"Describe a hot-air balloon for a curious child" is — the rubric
only cares that the response is well-organised, age-appropriate,
and answers the prompt as posed.

For `on_policy_rkl`, only the rollout matters — RKL is the divergence
between teacher and student rollouts, no semantic grading. Procedural
topics work identically.

### 2. Procedural function names in coding benches

`debug_bench` / `correction_bench` / `refactor_bench` previously had
8 / 8 / 3 fixed function names (`sum_to`, `first_minus_second`, etc.).
A miner could memorise patterns keyed on those names. v29.6 appends
a procedural 1-syllable suffix to each entry-point per item:

```
sum_to            → sum_to_pain
first_minus_second → first_minus_second_spays
at_least           → at_least_fro
find_largest       → find_largest_trien
mod_then_double    → mod_then_double_prail
product_of_list    → product_of_list_flest
first_or_default   → first_or_default_pear
second_last        → second_last_hain
count_evens        → count_evens_pien
sum_of_squares     → sum_of_squares_clurt
flatten_two_level  → flatten_two_level_grould
```

Tests call `candidate(...)` so they don't reference the function name
and need no rewriting; only the `def <name>(` sites in the buggy
code, the corrected signature, and (for correction_bench) the
simulated error-trace's "in <funcname>" line need consistent renaming.
The sandbox grader's `entry_point` argument carries the rotated name
through unchanged.

## What's still semi-static (and why)

Tiny fixed real-word lexicons remain inside the chat-prompt
synthesiser:

* **`_CHAT_AUDIENCES`** (12 entries) — needed so the model knows the
  register/tone to use ("a curious child" vs "an experienced engineer").
* **`_CHAT_ASPECTS`** (12) — needed for `compare` prompts to make sense.
* **`_CHAT_LANGUAGES`** (14) — translation requires real target languages.
* **`_CHAT_STYLES`** (7) — rewrite requires real style adjustments.
* **`_CHAT_TASKS`** (12) — howto/steps require real, achievable tasks.
* **`_CHAT_CATEGORIES`** + **`_CHAT_QUALIFIERS`** + **`_CHAT_DILEMMAS`**
  + **`_CHAT_VERB_OBJECTS`** + **`_CHAT_PLACES`** — same, needed as
  semantic anchors so the prompt is interpretable.
* **`_LEX_CONS` / `_LEX_VOWELS` / `_LEX_CODAS`** (37/12/12) — phoneme
  inventories for procedural pseudo-words. The COMBINATORIAL space
  (5300 syllables × N positions) is effectively unbounded; the
  inventory size is bounded for grammaticality.
* **debug/correction/refactor TEMPLATE BODIES** — each kind has a
  fixed bug pattern (off-by-one, swap-subtract, etc.) but the
  PARAMETERS (numbers in tests, threshold values, modulus values)
  are RNG-derived per item. The bug PATTERN must be fixed because
  it's what the model is being tested on; what rotates is the surface
  identifier (function name, parameter values).

These remaining static elements are necessary scaffolding — replacing
them with pure pseudo-words would break the model's ability to
interpret the prompt or yield meaningless rubric grades. The
combinatorial composition with procedural slots makes the
prompt-LEVEL surface form effectively uncacheable, which is the
property we actually need.

## Forward path (still pending)

* **Per-round procedurally generated bug patterns** for
  debug/correction/refactor — currently the parameter values rotate
  but the BUG PATTERN per kind is fixed. Could parameterise the
  bug-injection step itself (e.g. multiple variants of off-by-one:
  `range(1,n)`, `range(0,n)`, `range(2,n+1)`). Defer until per-axis
  saturation telemetry shows the patterns memorising.
* **Expanded `_CHAT_*` lexicons** — could grow each from 10-15 entries
  to 50+ entries to slow lexicon-level memorisation. Combinatorial
  expansion already makes the prompt-level surface uncacheable but
  this is a low-cost extra layer.
* **Procedural language pool** — currently 14 fixed real languages.
  Could expand to 50+ to match the world's major language inventory.
