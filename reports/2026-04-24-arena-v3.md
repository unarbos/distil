# Arena v3 — comprehensive eval for SOTA small-model distillation

**Author:** distil-validator maintenance rotation
**Status:** Shipping 2026-04-24. Session 2 axes promoted to production
ranking immediately (48h shadow lead-time already served). Session 3 axes
shipped in shadow mode, promoted to production ranking and Pareto
dominance becomes a dethrone gate on 2026-04-26 (+48h notice).
**Supersedes:** `reports/2026-04-24-pareto-holistic-eval-v2.md` (v2 is
now live). Extends `reports/2026-04-23-goodhart-immune-eval.md`.
**Inspiration:** [AffineFoundation/affine-cortex](https://github.com/AffineFoundation/affine-cortex)
— Bittensor Subnet 120. We adopt the pattern (Pareto dominance across
many held-out absolute-correctness axes, distillation as one axis among

## 1. The 2026-04-23 recap — what Session 2 solved and what it didn't

Session 1 (2026-04-23) closed the obvious KL-Goodhart holes but every
axis was still *relative* — scored against the teacher or normalized
against the king. Session 2 added five absolute-correctness bench axes
(`math_bench`, `code_bench`, `reasoning_bench`, `knowledge_bench`,
`ifeval_bench`) drawn from GSM8K / MATH-500 / HumanEval / BBH /
MMLU-Pro / IFEval. Those axes have been in shadow since 2026-04-24 and
now power the live leaderboard. Overfitting any of them now translates
directly into a measurable SOTA capability — not a relative claim against
the teacher.

What Session 2 still leaves open:

1. **Capability ceiling is too low.** GSM8K and MATH-500 are ~90%-
   solvable by GPT-4. A miner that overfits them plateaus at a
   grade-school-math model. We need at least one axis that goes past
   the teacher's own frontier.
2. **No agentic signal.** The teacher knows how to use a calculator;
   the student will not, unless the eval rewards tool use. Most real
   SOTA small-model work (Phi-4, Qwen2-Math, DeepSeek-R1) has a tool-use
   leg.
3. **One-shot luck vs real knowledge.** Every current axis takes the
   first greedy sample. A model that half-knows an answer and happens
   to sample the right token wins the same as a model that reliably
   knows. Self-consistency (sample many, majority-vote) separates the
   two.
4. **Pareto gate is missing.** The Session 2 worst-axis floor is a
   defense against *collapse* on a single axis. A challenger that beats
   KL by 3% but is worse than the king on a majority of capability
   axes still passes today. Affine's key idea — Pareto dominance — is
   the right gate for "genuinely better, not just KL-better."

### 2.1 `aime_bench` — olympiad math (AIME25 + AIME2024 + aimo-validation)

* ~90 olympiad-level math items from three public sources, all cached
  offline on the eval pod via `HF_TOKEN`.
* Prompt format: "Solve step-by-step. End with `\\boxed{...}`." Scoring
  extracts the boxed integer (with robust `\\frac{}{}` / `\\text{}{}`
  stripping) and compares to the gold integer.
* Per-round budget: `BENCH_AIME_PER_ROUND=4`, 1536-token cap each.
* Why: raises the capability ceiling past teacher-level. AIME is hard
  enough that the teacher itself scores in the 15-40% range; a student
  that clears 45% is doing something real.
* Seeded sampling: `_pick_bench_items` keyed off `block_seed` picks the
  same 4 items for every validator but rotates every round → zero
  memorization payoff.

### 2.2 `mbpp_bench` — MBPP+ coding

* 378 programming items (function signature + docstring + test list),
  sandboxed in the existing `humaneval_sandbox.py` subprocess.
* 3s timeout per test list. Pass-fraction = items passing all tests.
* Per-round budget: `BENCH_MBPP_PER_ROUND=2`.
* Why: HumanEval alone is narrow (single-function, short). MBPP+
  covers a much wider problem distribution and has a formally-verified
  test suite. Two axes testing code means a miner that overfits one
  still has to generalize to the other.

### 2.3 `tool_use_bench` — agentic Python calculator

* Numerically-tractable items sampled from `math_pool`.
* First pass: prompt the student with a system message telling it it
  can write `<python>...</python>` to run Python. Student response is
  parsed; the first `<python>` block is extracted.
* The extracted snippet is executed via `humaneval_sandbox._run_in_subprocess`
  with a 3s timeout, no network, no filesystem writes. The captured
  stdout (`_BENCH_TOOL_USE_STDOUT_CAP=256` chars) is the tool result.
* Second pass: original prompt + assistant's first pass + tool result
  is re-fed to the model for a final `\\boxed{}` answer.
* Scored: integer/float answer compared to gold (±1e-3 tolerance).
* `tool_used_count` is exposed on the axis payload so we can track how
  often miners actually reach for the tool vs solving direct.
* Why: this is the crucial Phi-4 / Qwen2-Math capability. A small model
  that knows when to defer arithmetic to a calculator is strictly
  better than one that fabricates.

### 2.4 `self_consistency_bench` — majority-vote reliability

* Hard-math items (filtered on gold length > 2 digits) from
  MATH-500.
* K=5 samples at `BENCH_SELF_CONSISTENCY_TEMP=0.7`,
  `BENCH_SELF_CONSISTENCY_TOPP=0.9`. Each sample uses a deterministic
  seed derived from `(block_seed, item_id, k)`.
* Majority-vote on extracted `\\boxed{}` answers. Tie breaks on the
  first winning answer encountered.
* Per-round budget: `BENCH_SELF_CONSISTENCY_PER_ROUND=2` (each item
  expands to 5 samples → ~10 generations).
* Why: separates "happens to greedy-sample right" from "really knows."
  A model that is 60% right greedy and 85% right after 5-sample
  majority vote is a fundamentally more reliable model than one that
  is 80% right greedy but 75% right after majority vote (an unstable
  answer distribution).

### 2.5 `arc_bench` — commonsense science MC (Session 3.1, 2026-04-25)


* AI2 ARC-Challenge (1172 items, `allenai/ai2_arc` / `ARC-Challenge`
  split, test). Multiple-choice science questions graded for
  "Challenge" difficulty by AI2 (i.e. items where both a retrieval-
  based and word-cooccurrence solver both failed).
* Prompt format identical to `knowledge_bench`: question + labeled
  options + "Respond with only the letter of the correct answer".
* Answer-letter extraction via the same `_MMLU_LETTER_RE` regex so the
  scoring pipeline is shared. Raw `answerKey` values of `"1"`/`"2"`/
  `"3"`/`"4"` are auto-mapped to the corresponding A/B/C/D label.
* Per-round budget: `BENCH_ARC_PER_ROUND=6`, 48-token cap each.
* Why: ARC is authored by science educators with a different intent
  than MMLU (pedagogical / deliberately tricky), and the Challenge
  split is specifically filtered to questions that simple solvers
  fail. Climbing this axis independently of `knowledge_bench`
  demonstrates commonsense-science reasoning that isn't just MMLU-
  memorized.
* Status: SHADOW on 2026-04-25 ship. The same +48 h promotion path as
  the rest of Session 3 (there is no separate gate flag; `arc_bench`
  lives in `ARENA_V3_AXIS_WEIGHTS`, so flipping
  `ARENA_V3_AXES_IN_COMPOSITE=1` on 2026-04-26 promotes all of them
  together).

### 2.6 `reasoning_density` — length-efficient correctness (Session 3.2, 2026-04-25)

User-reported pathology: *"models are too distilled and think for too long
about simple questions."* The existing `length` axis scored the chat probe
only, so long `<think>` blocks on trivia questions in our **bench** probes
were invisible. Fix:

* Every bench probe now records per-item `gen_tokens` (token count of the
  generated output before stop) and, at finalization,
  `mean_gen_tokens` / `mean_gen_tokens_correct` (mean over items /
  correct-only items). See `_bench_finalize_token_stats` in
  `scripts/pod_eval_vllm.py`.
* `_axis_reasoning_density(student)` computes, per bench with a valid
  `mean_gen_tokens_correct`, `pass_frac × length_bonus`. Then averages
  across benches.
* `length_bonus = 1.0` if `mean_gen_tokens_correct ≤ target`, and
  `1.0 / (1.0 + (ratio − 1.0))` above target.
* `REASONING_DENSITY_TARGET_TOKENS` targets are empirically calibrated to
  the teacher's typical bench output lengths — a student hitting teacher
  length (or shorter while correct) scores 1.0.

Overfit-resistance argument: a model that short-circuits and outputs
"A" (one token) on every `knowledge_bench` item has `pass_frac ≈ 0.25`
(random), so `reasoning_density ≈ 0.25`. To get a high density score the
model has to BE correct AND concise — exactly the deployment profile
users want. The "memorize answer-only" failure mode and the
"verbose-rambling" failure mode are BOTH penalized.

Per-bench targets (tunable via `RD_*_TARGET`):

| Bench              | Target (tokens) | Rationale                                                |
|--------------------|----------------:|----------------------------------------------------------|
| `math_bench`       | 400 | GSM8K+MATH with CoT but not rambling.                           |
| `code_bench`       | 300 | HumanEval function body, moderate comments.                     |
| `reasoning_bench`  | 150 | BBH exact-match; most items have short answers.                 |
| `knowledge_bench`  | 30  | MC letter + optional short justification.                       |
| `ifeval_bench`     | 250 | Format-following responses vary by prompt.                      |
| `aime_bench`       | 800 | Olympiad problems need long CoT by design.                      |
| `mbpp_bench`       | 250 | Function body at MBPP difficulty.                               |
| `tool_use_bench`   | 300 | One Python call + final answer.                                 |
| `self_cons_bench`  | 300 | CoT math; 5 samples averaged.                                   |
| `arc_bench`        | 50  | Grade-school science MC; letter + brief justification.          |

* Weight: `REASONING_DENSITY_WEIGHT=0.05`, shadow by default.
* Gate: `REASONING_DENSITY_IN_COMPOSITE=1` promotes.
* Version bump: `COMPOSITE_SHADOW_VERSION=6`.

### 2.7 `chat_turns_probe` — multi-turn coherence (Session 3.3, 2026-04-25)

Single-turn `judge_probe` captures "answers one question well"; it does
NOT test whether the model can hold context across turns. A distilled
model can ace KL yet still repeat itself, forget what it just said, or
contradict itself the moment the user asks a follow-up. That failure
mode is invisible to every axis we ship today.

Design mirrors `judge_probe` exactly so the normalization is directly
comparable:

* **Pool:** 24 hand-authored 3-turn conversations in
  `CHAT_TURNS_PROBE_POOL`. Each tuple is `(user_1, user_2, user_3)`.
  Topics span dinner planning, running plans, hash tables, book recs,
  plant care, debugging, weekend trips, guitar, haiku, JavaScript
  scoping, TDD arguments, Python/Rust trade-offs, sugar cravings,
  Hamlet consistency, quantum computing, résumés, interest rates,
  cookie baking, meditation, Dijkstra's algorithm, career pivots,
  rainy-Saturday kids, binary heaps, and interview nerves — deliberately
  broad so memorizing templates is unhelpful.
* **Rotation:** 6 per round via `block_seed` with a distinct mixer
  (`0xBF58476D1CE4E5B9`) so it doesn't phase-lock with `judge_probe`.
* **Phase A:** student generates an assistant turn after each user turn,
  with the accumulated transcript as context. 200-token cap per turn,
  `enable_thinking=False`, greedy. 3 × 6 = 18 generations per student
  (~3–4 s total).
* **Phase B:** teacher receives the full transcript and grades it on a
  1–5 rubric focused on coherence + consistency + helpfulness:
  - 5 = excellent on all three criteria, reads like a thoughtful human
  - 4 = good; minor issues on at most one criterion
  - 3 = mediocre; forgets context or gives vague answers
  - 2 = poor; often unhelpful, inconsistent, or off-topic
  - 1 = contradictory, nonsensical, or unrelated to user turns
* **Normalization:** `(mean_score − 1) / 4`. Identical shape as
  `judge_probe.normalized`.
* **Axis:** `_axis_chat_turns_probe(student)`. Drops to None if
  `n_valid < CHAT_TURNS_MIN_VALID=3` (rubric parse failure protection).
* **Weight:** `CHAT_TURNS_AXIS_WEIGHT=0.08`, shadow by default.
* **Gate:** `CHAT_TURNS_AXIS_IN_COMPOSITE=1` promotes.
* **Version bump:** `COMPOSITE_SHADOW_VERSION=7`.

Overfit-resistance argument: the rubric is holistic and depends on
multi-turn reference (the user's turn 3 explicitly depends on the
assistant's turn 2 in most conversations, e.g. "Based on what I said,
what is the single first song I should learn?"). A model can't
short-circuit this with memorized one-liners; it has to actually
carry context. Optimizing this axis ⇒ better deployment chat quality,
not a narrower skill.

### 2.8 `truthful_bench` — adversarial factuality (Session 3.4, 2026-04-25)

Every other absolute-correctness axis tests *capabilities* (can you
solve math? can you write code?). None of them directly test whether
the model prefers facts over plausible-sounding falsehoods. Pure
distillation inherits the teacher's priors — including the same
hallucinations the teacher has. A SOTA small model should lose these;
today we have no axis that rewards that loss.

`truthful_bench` is the axis that rewards it.

* **Dataset:** `truthful_qa` config `multiple_choice`, split
  `validation`, 817 items (pre-cached on the pod). Each item is an
  adversarial factual question ("What happens if you swallow gum?")
  with K candidate answers (K ∈ [2, 13], mean 5) where exactly one is
  labelled correct in `mc1_targets`. Wrong answers are specifically
  chosen to match popular misconceptions.
* **Anti-gaming shuffle:** the raw dataset always places the correct
  answer at index 0 (verified empirically: 817/817 items). Iterating
  options in order lets a model answer "A" on every item and score
  100%. We shuffle the K choices deterministically per item using
  `SHA-256(question)[8:16]` as the RNG seed, so the correct letter
  varies item-to-item but every validator produces the same prompt
  for the same question. The empirical gold distribution after
  shuffle is A: 21.7%, B/C: 20.4%, D: 17.6%, E: 10.2%, F: 5.9%, G–J
  rare — well-mixed given the K distribution.
* **Item truncation:** items with K > 10 are trimmed to 10 options
  (correct answer + 9 deterministically-sampled wrong options) so the
  output letter fits A–J. Four items in the pool have K > 10.
* **Rotation:** 4 per round via `block_seed` with stream
  `0x74717A01`, disjoint from every other axis.
* **Probe:** `truthful_bench_probe`. Same shape as
  `arc_bench_probe` — render MC prompt, generate ≤48 tokens with
  `enable_thinking=False`, extract first A–J letter, compare to gold.
* **Axis:** `_axis_truthful_bench(student)` → `_axis_bench_pass_frac`.
  `BENCH_MIN_VALID["truthful_bench"] = 2`.
* **Weight:** `BENCH_TRUTHFUL_WEIGHT=0.03` (conservative because the
  MC framing is slightly noisier than gold-text matching — 10% of
  items have contentious "correct" labels per recent TruthfulQA
  critiques, so we under-weight).
* **Gate:** rolls into `ARENA_V3_AXES_IN_COMPOSITE` with the rest of
  Session 3.
* **Version bump:** `COMPOSITE_SHADOW_VERSION=8`.

Overfit-resistance argument: the only way to climb is to add factual
grounding training data (TriviaQA-factual, RefuseElseFalse,
HaluEval-sft, TruthfulQA train). A model trained on these is strictly
more useful in deployment — hallucination is the #1 reported failure
mode of small models. The per-item shuffle + 817-item pool rotation
+ 4-per-round sampling means memorizing the answers is infeasible
(you'd need to see ~200 rounds of the same pool before covering it).

### 2.9 `long_context_bench` — procedural needle-in-haystack (Session 3.5, 2026-04-25)

Every other absolute-correctness axis has a fixed public dataset —
even with rotation, memorization is theoretically possible if a miner
is patient enough. And every other axis lives in the under-800-token
range. `long_context_bench` closes both gaps in one stroke:

* **Procedural generation:** there is no dataset. Every round we
  generate 3 fresh items from `block_seed` — a unique question, a
  unique secret-answer needle, and a distractor document (40 lines,
  ~600 tokens default; bumpable via `BENCH_LC_DISTRACTORS`). The
  test set literally does not exist outside the live round, so
  memorization is impossible at the item level. A miner can only
  climb by getting genuinely better at long-context retrieval.
* **Content:** each item is:
  * A bank of 40 generic "filler" distractor sentences (e.g. "The
    lecture hall echoed with polite applause", "A curl of steam rose
    from the untouched coffee").
  * One needle sentence from 20 templates (e.g. "The secret passcode
    was `{X}`", "Aria's favorite number was `{X}`") with `{X}` drawn
    from a random codeword pool (color-animal, phonetic alphabet,
    numeric sequences).
  * The needle is inserted at a random position in the distractors.
  * The question asks specifically for `{X}`.
* **Prompt:** context document + "Answer with only the exact value,
  no explanation." Generated with `enable_thinking=False` and
  `max_tokens=32` so the student must answer directly, not reason.
* **Scoring:** case-insensitive substring containment of the gold
  needle in the output. Lenient on formatting (accepts "The passcode
  is blue-falcon-7", "blue-falcon-7", "It's blue-falcon-7") while
  still catching hallucinations — the probability of
  randomly-matching the unique needle across our templates is
  <0.1%.
* **Rotation:** 3 per round via `block_seed` with stream
  `0x10C0001D`, disjoint from every other axis.
* **Probe:** `long_context_bench_probe`. Same vLLM generation path as
  every other absolute-correctness bench.
* **Axis:** `_axis_long_context_bench(student)` →
  `_axis_bench_pass_frac`. `BENCH_MIN_VALID["long_context_bench"] = 2`.
* **Weight:** `BENCH_LC_WEIGHT=0.03`. Conservative for now while we
  gather first-round data; can bump once miners respond.
* **Gate:** rolls into `ARENA_V3_AXES_IN_COMPOSITE` with the rest of
  Session 3.
* **Version bump:** `COMPOSITE_SHADOW_VERSION=9`.

Overfit-resistance argument: this is the strongest Goodhart-immune
axis in the system. Since the pool is regenerated every round from a
new seed, there is no training target — a miner cannot pre-compute
or memorize anything. The only way to climb is to train on
general-purpose long-context retrieval data (RULER, NeedleBench,
long-context SFT from books/wikipedia). A model that scores well
here is unambiguously better at deployment-relevant retrieval.
Additionally, the axis forces miners to preserve long-context
attention — a capability that aggressive quantization and LoRA-only
training often breaks.

## 3. Pareto-dominance dethrone gate

New function `composite.compute_pareto_dominance(challenger_axes,
king_axes, margin=0.02)` returns:

* `wins`: axes where the challenger beats the king by more than
  `margin` (default 2% absolute, tunable via
  `PARETO_DOMINANCE_MARGIN`).
* `losses`: axes where the king beats the challenger by more than
  `margin`.
* `ties`: axes within the margin (treated as noise).
* `pareto_wins` (bool): challenger wins a majority of comparable axes
  AND `n_wins >= n_losses`. This is *soft* Pareto — strict Pareto is
  unwinnable at small K.

The gate:

1. Runs on both the paired-t-test and legacy-epsilon dethrone paths.
2. Starts in **shadow mode** (`PARETO_DOMINANCE_GATE=0`). Logged and
   surfaced in telemetry so miners can see where they stand, but does
   NOT block dethronement yet.
3. Flips to active block on **2026-04-26** (+48h notice on Discord).
4. Fails open if fewer than `PARETO_DOMINANCE_MIN_COMPARABLE` (default
   5) axes have data on both sides — pod probe outages can't freeze
   the crown.

The semantics: a challenger that beats KL by 3% but is worse than the
king on `math_bench`, `code_bench`, `reasoning_bench`, `judge_probe` is
refused the throne. The existing worst-axis floor stays in place; the
Pareto gate is additive.

## 5. What overfitting these axes produces

The claim "overfit our eval → SOTA small model" requires each axis to
map to a genuine capability. Walking the list:

| Axis                     | If a miner optimizes this, they get… |
|---------------------------|-------------------------------------|
| `kl`, `on_policy_rkl`    | A model close to the teacher distribution — SOTA-adjacent at teacher's ceiling. |
| `capability`              | A model that solves the teacher's verifiable prompts independent of length. |
| `length`                  | A model that doesn't ramble — fast inference. |
| `degeneracy`              | A model that doesn't collapse into loops at long context. |
| `judge_probe`             | A model the teacher rates as helpful on realistic prompts. |
| `math_bench`              | Grade-school + high-school math. |
| `code_bench`              | Function-level Python. |
| `reasoning_bench`         | BBH multi-step reasoning. |
| `knowledge_bench`         | MMLU-Pro world knowledge. |
| `ifeval_bench`            | Instruction-following under format constraints. |
| `aime_bench`              | Competition-level math reasoning. |
| `mbpp_bench`              | Broader, semantically-varied code. |
| `tool_use_bench`          | **Agentic:** knows when to use the Python tool. |
| `self_consistency_bench`  | **Reliable:** multi-sample consistency. |
| `arc_bench`               | Commonsense science reasoning (pedagogically-authored). |
| `truthful_bench`          | **Hallucination-resistance** — model prefers facts over plausible-sounding falsehoods. The #1 user-reported failure mode of small models. |
| `long_context_bench`      | **Needle-in-haystack retrieval** — the model actually reads the context window you give it instead of leaning on its priors. Procedural generation makes this the most Goodhart-immune axis we ship. |
| `reasoning_density`       | **Concise AND correct** — no long CoT on trivia, terse-but-wrong also penalized. The deployment profile users actually want. |
| `chat_turns_probe`        | **Multi-turn coherence** — the model that can hold a conversation. |

Composite `worst` is the minimum across these (after shadow promotion).
A miner has to make every single one >= `COMPOSITE_DETHRONE_FLOOR` or
it cannot take the crown. Pareto dominance adds that the miner has to
win *a majority* — not just "be equally good, slightly closer on KL."

The only way to climb is to broadly get better at small-model
intelligence. That's what we wanted.

## 6. What we explicitly did NOT do

* **SWE-bench / browser tasks / MCP servers / OpenSpiel.** Affine has
  these; we don't. They require docker-in-docker or dedicated sidecar
  infra, which isn't available on a single eval pod. Adding them later
  would require a refactor of the pod/rendezvous model and is out of
  scope for this ship. Our tool-use axis is the narrow in-pod analog.
* **GPQA-Diamond.** The dataset is gated on Hugging Face even with an
  authenticated token. We skipped it rather than ship a probe that
  silently errors. 2026-04-25 update: we added `arc_bench`
  (AI2 ARC-Challenge, 1172 items, cached) in Session 3.1 as a
  replacement commonsense-science axis — see §2.5 below.
* **Cross-model peer comparisons.** Affine's ELO adapts across past
  models. We keep single-round comparisons for determinism and to
  avoid "king shift" destabilizing the score over time.
* **Full ground-truth cross-validation.** We use deterministic
  per-round seeded sampling but do not hold out a separate
  development set. A miner can, in principle, inspect past seed
  rotations and memorize the union. The per-round sample budget
  (2-4 items × 9 axes = ~30 items) keeps the cumulative set small
  enough that memorization is harder than learning, but that is a
  probabilistic argument, not a proof. If we ever see a model that
  nails exact-match on items far past what its KL would predict,
  we have the `reports/2026-04-23-goodhart-immune-eval.md` tripwires
  to detect it.

## 9. Open questions / next sessions

1. **Tool-use budget.** We cap Python execution at 4s. If miners start
   submitting expensive sandboxed computations (e.g. integer
   factorization), raise `BENCH_TOOL_USE_SANDBOX_TIMEOUT_S` carefully
   to keep per-round wall time stable.
2. **Self-consistency K.** K=5 is a conservative starting point. If
   per-round budget allows, K=10 is materially more discriminative.
3. **GPQA-Diamond access.** Re-try with `hf auth request` for the
   gated dataset. If we get access we can add `gpqa_bench` as a 10th
   axis for graduate-level reasoning.
4. **ELO-style cross-round ranking.** Once Pareto is live, start
   collecting per-axis deltas into an ELO table for the leaderboard.
   Useful for showing miners "you beat the king on math, lost on code
   — by how much, over how many rounds."
5. **Agentic environments beyond tool-use.** Browser / filesystem /
   multi-turn chat. Would require a sidecar. Defer until the single-
   pod rig is saturated (currently at ~80 min/round wallclock).

Shipping this as one ship with a 48h promotion window for the gates.
If the new axes introduce any pod instability, we can individually
flip them off via `BENCH_*_PER_ROUND=0` without reverting code.
