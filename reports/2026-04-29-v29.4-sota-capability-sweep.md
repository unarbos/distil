## TL;DR

Implements 5 of 6 prioritized roadmap items in one commit, plus
capability axis tightening:

1. **`capability` tightening** — drop static trivia 12→4, bump
   procedural 24→32, add 7 harder kinds.
2. **`correction_bench`** — buggy code + explicit error trace → fix.
3. **`multi_doc_synthesis_bench`** — 4 procedural fact cards +
   cross-card question (sum / difference / compare / ratio).
4. **`calibration_bench`** — solvable + unsolvable mix; reward
   correct refusals, penalise confabulation.
5. **`refactor_bench`** — working code + style constraint;
   AST-graded constraint check on top of behaviour tests.
6. **Teacher-swap readiness** — `ACTIVATION_FP_VOCAB_SIZE` env-

   from `subnet-config.json` instead of hardcoded "Qwen3.5".

needs teacher-loading integration that should land alongside the
teacher swap so we don't have to redo the work.

## Why each axis

| Axis | Capability gap it closes |
|------|---------------------------|
| `capability` (tightened) | Saturation on a 0.25-weight axis was killing discrimination at the top of the leaderboard. New hard kinds (multi-step arithmetic, sequence pattern recognition, multi-step string/list ops, counterfactual reasoning, nested boolean logic, two-step compare) test compositional reasoning rather than single-step lookups. |
| `correction_bench` | Real coding workflow includes reading an error message and applying a targeted fix. `code_bench` (write-from-scratch) and `debug_bench` (find AND fix) don't measure this specific skill. |
| `multi_doc_synthesis_bench` | `long_context_bench` tests retrieval within ONE document. Real SOTA models also synthesise across multiple discrete documents — sum / compare / ratio across distinct fact sources. |
| `calibration_bench` | SOTA models say "I don't know" when uncertain. No current axis catches confabulation; a model that produces fluent-but-wrong answers passes our other axes. This binds hard against that failure mode. |
| `refactor_bench` | SOTA models can restructure existing code while preserving behaviour. Distinct from write-from-scratch and bug-fix. AST-graded constraint enforcement (no nested loops / max lines / no explicit loop) prevents shortcut answers. |

## Capability tightening — concrete change

Before:

```
CAPABILITY_PROBE_N = 12        # static trivia
CAPABILITY_PROBE_N_PROC_MATH = 24  # 15 easy procedural kinds
```

53 % of records saturated at 1.0; mean 0.90. The legacy procedural
kinds (add/sub/mul/div/mod, count_chars, list_min, etc.) were too easy
for 4B-class models.

After:

```
CAPABILITY_PROBE_N = 4         # smaller trivia floor, kept for format diversity
CAPABILITY_PROBE_N_PROC_MATH = 32  # 22 kinds: 15 easy legacy + 7 new hard
```

New hard kinds:
- `multi_step_arithmetic` — `(a+b) * c - d`
- `seq_next` — Fibonacci-like / arithmetic / geometric / square sequences
- `string_chain` — reverse + uppercase + count vowels (multi-step string ops)
- `list_chain` — filter even + sum + multiply by 2 (multi-step list ops)
- `counterfactual` — "if x were Y instead of X, what would x*y be?"
- `nested_logic` — boolean expression with negation + parens
- `two_step_compare` — compute two values then compare

Expected effect: mean drops to ~0.55-0.65, saturation drops to ~15 %,
restoring discrimination on the 0.25-weight axis.

## New axes — design notes

### `correction_bench`

Reuses the 8 bug kinds from `debug_bench` but adds an explicit
simulated error trace (deterministic per kind + parameters, no
sandbox call at generation time so it stays microsecond-fast and
validator-agreement-deterministic). Item shape matches
`code_bench` so `humaneval_sandbox` runs unchanged.

Sample prompt structure:

```
# The following Python function fails its tests with the
# error trace shown below. ...
#
# Buggy version (DO NOT include in your output):
# def sum_to(n):
#     total = 0
#     for i in range(1, n):
#         total += i
#     return total
#
# Test failure trace:
#   File "test_sum_to.py", line 3, in test_sum_to
#     assert candidate(8) == 36
#   AssertionError: expected 36, got 28 (off by 8)
#
# All tests the corrected version must pass:
# assert candidate(8) == 36
# assert candidate(1) == 1
# ...
#
# Now complete the corrected version below.
def sum_to(n: int) -> int:
    """Return the sum of integers from 1 to n inclusive. ..."""
```

Model writes the body. Sandbox runs `prompt + gen + test`; the
buggy version is in comments so it never executes.

### `multi_doc_synthesis_bench`

4 procedural fact cards each describing a unique fictional
organisation (the Iversfeld Observatory, the Kestrel Beekeepers'
Union, etc.). Each card has a numeric attribute drawn from
non-overlapping ranges (card[0] ∈ [100,180], card[1] ∈ [300,380], …)
so substring grading sees only one numeric match for any given gold.

Question kinds:
- `sum`: "what is the COMBINED total of A and B's attribute?"
- `difference`: "how many more does A have than B?"
- `compare`: "which has more, A or B?" → gold is the topic name; confusers are the OTHER topic names so the model must commit
- `ratio`: "how many times larger is A's attribute than B's? (integer)"

Substring grading + confuser rejection (same shape as
`long_context_bench` v29.2 multi-needle items).

### `calibration_bench`

Mix of solvable and unsolvable arithmetic-narrative items. Templates
deliberately render TWO versions:

```python
solv:   "X owns 8 books in the kitchen and 9 books in the study. ..."
unsolv: "X owns books in the kitchen and 9 books in the study. ..."
```

Solvable items have a numeric gold; the model passes if the gold
appears AND no refusal token appears (so a model that hedges to
get free credit fails). Unsolvable items have `gold = REFUSE`; the
model passes if a refusal token appears AND no standalone integer
appears (so a model that says "I don't know but if I had to guess,
42" fails).

This dual gate forces calibration: a model that always confabulates
fails the unsolvable half; a model that always refuses fails the
solvable half. Only a calibrated model passes both.

Per-half pass-fracs are surfaced in telemetry (`solv_pass_frac`,
`unsolv_pass_frac`) so we can see which mode a model is failing.

### `refactor_bench`

Working but stylistically poor reference function (nested loops,
spurious while-inside-for, etc.) + a style constraint:

- `no_nested_loops` — no for-inside-for, while-inside-while, etc.
- `max_lines` — refactor body ≤ N source lines
- `no_explicit_loop` — must use comprehension / `sum` / `map` / etc.

Grader runs in two stages:
1. `humaneval_sandbox.run_batch` checks behaviour (all asserts pass).
2. AST analysis on the model's emitted function checks the constraint.

Item-level pass requires BOTH. A "correct but ugly" refactor fails
on the constraint; a "clean but wrong" refactor fails on tests.

The path to "models that overfit our eval = SOTA-quality 4B models"
runs through these axes. After v29.4 the validator scores miners on:

* **Reasoning** — math (gsm8k-style), code (humaneval-style),
  reasoning (BBH-style), aime (olympiad), mbpp, capability (multi-step)
* **Coding** — write-from-scratch (code/mbpp), find-and-fix (debug),
  parse-and-fix (correction), refactor (refactor), tool use
* **Comprehension** — long-context single-doc retrieval +
  multi-doc synthesis, ifeval (structured following)
* **Robustness** — paraphrase / noise (robustness), calibration
  (refusal vs confabulation)
* **Distillation** — KL, on-policy RKL, capability, length,
  degeneracy, judge_probe, chat_turns_probe, reasoning_density

A miner who is Pareto-strong across all of those is, by construction,
producing a 4B-class SOTA model. The composite.worst rule + per-axis
baseline-relative penalty + Pareto majority gate together make that
the only winning strategy.
