# 2026-04-29 v29.2 — debug_bench + long_context fix + correlation telemetry

**Status:** SHIPPED 2026-04-29 00:30 UTC. Three coordinated changes that
keep moving the validator toward "models that overfit our eval are the
most performant SOTA open-weights models."

## TL;DR

The 2026-04-28 post-v29.1 audit on 163 UIDs surfaced three concrete
gaps:

1. **`long_context_bench` was 93 % saturated** (mean 0.94 across all
   records). With only 3 confusers and ~14-line documents, every
   4B-class model trivially identified the entity-matched needle —
   the axis was a free 0.94 contribution to composite for every
   miner, contributing zero discrimination at the top of the board.
2. **`debug_bench` capability gap.** SOTA coding requires both
   write-from-scratch (which `code_bench` / `mbpp_bench` measure) AND
   debugging existing code. A miner could ace those two axes while
   being unable to read+fix a buggy function, and we'd never know.
3. **No real-time correlation telemetry.** We knew kings were
   regressing on the held-out canary, but we didn't have a per-axis
   drift signal. First run of the new
   `scripts/audit/axis_correlation.py` revealed pre-v29 `math_bench`
   correlation with held-out gsm8k of **-0.875** (n=5) — a smoking-gun
   Goodhart signature.

v29.2 closes all three:

1. **`long_context_bench` v29.2.** Multi-needle items (40 % of round)
   require retrieving 2-3 distinct needles and assembling them into a
   hyphenated composite gold. Confusers bumped 3 → 6, distractors
   40 → 60. The hyphenated form preserves substring grading
   correctness (no individual code contains a hyphen, so a model that
   happens to mention a confuser code while reasoning still gets
   rejected only on the confuser, not on the gold itself).
2. **`debug_bench` axis.** Procedural buggy-code items, 8 bug kinds
   (off-by-one, swap operands, wrong comparator, wrong init, early
   break, wrong index, wrong modulo, missing edge case). Same item
   shape as `code_bench` so the existing `humaneval_sandbox` runs
   unchanged — auto-indent / prose-trim / fence-strip all apply
   identically. Joins `BASELINE_RELATIVE_PENALTY_AXES` so a model
   regressing on debugging vs Qwen-4B-base loses ranking
   proportionally.
3. **`scripts/audit/axis_correlation.py`.** Computes per-axis Pearson
   r between each procedural axis and its held-out counterpart, joined
   by king UID. Writes `state/axis_correlation.json` for dashboard
   consumption. Cheap (~1 second on 50 rounds), runnable from a cron
   or post-round hook.

## Expected effects

### Immediate (next 1-2 rounds)

* **`long_context_bench`** drops from 93 % saturation to ~50-65 % pass
  rate. Multi-needle items demand actual retrieval + assembly; a model
  that regex-extracts any 7-char code in the doc gets rejected by the
  confuser-rejection layer because all confusers also have 7-char
  codes. Single-needle items remain at 60 % of the round so the
  difficulty floor is preserved.
* **`debug_bench`** lands as a fresh axis at the bottom of every
  composite. Initial expectation: Qwen-4B-base scores 0.40-0.55,
  current kings score 0.30-0.55 (debug is harder than write-from-
  scratch for 4B models). The baseline-relative penalty fires
  immediately — any king regressing here loses ranking proportionally.
* **Per-axis correlation telemetry** will start tracking new paired
  data (post-v29 kings + their post-v29 evalscope canary). Within a
  week we should see whether v29 procedural math templates flip
  `math_bench`↔gsm8k from -0.875 to positive. If not, the script tells
  us instantly and we iterate the templates.

### 2nd-order (within 1 week)

* Miners optimising the composite now have CONCRETE incentive to:
  - Retain Qwen-4B-base's debugging capability (or improve on it).
  - Produce models that handle multi-needle long-context tasks, not
    just single-needle retrieval.
  - Stay above Qwen-4B-base on every transfer axis (math/code/reason/
    ifeval/aime/mbpp/tool_use/long_context/robustness/debug).
* The composite.worst rule + per-axis baseline-relative penalty +
  debug_bench together push toward a Pareto frontier where progress
  means **beating base on every axis**, not just gaming one or two.
  This is the structural form of "overfitting our eval = SOTA model."

## Risks + rollback

* **`long_context_bench` could swing too hard the other way.** If
  multi-needle items are too hard, pass-rate could drop to ~20 %
  instead of the target ~50-65 %. Mitigation: `BENCH_LC_MULTI_FRACTION`
  is env-controlled; lowering 0.4 → 0.2 brings the multi share down. We
  also retained the single-needle floor at 60 % of the round.
* **`debug_bench` might saturate fast** if 4B models nail the bug
  patterns. The `BENCH_DEBUG_WEIGHT=0.06` is conservative; we can
  raise it once saturation telemetry confirms it's discriminating.
  Conversely, if it's too hard (everyone scores 0), we'll see that
  in the next saturation audit and add easier bug kinds.
* **`humaneval_sandbox` could choke on the new prompt format.** Smoke-
  tested against 8 hand-written fixes — all 8 pass cleanly. The prompt
  ends mid-`def` block exactly like `code_bench`, so the auto-indent /
  prose-trim layer applies unchanged.

Single-flag rollback per axis:

```bash
echo "BENCH_LC_MULTI_FRACTION=0.0" >> /home/distil/.secrets/distil.env  # back to single-needle only
echo "BENCH_DEBUG_PER_ROUND=0"      >> /home/distil/.secrets/distil.env  # disable debug_bench
echo "BENCH_DEBUG_WEIGHT=0.0"       >> /home/distil/.secrets/distil.env  # mute composite contribution
```
