# 2026-04-29 v29.5 — procedural everywhere: lex synthesis + zero static pools

**Status:** SHIPPED 2026-04-29 02:55 UTC. The user pushback after v29.4
was "we should have nothing static at all." This commit drives all
in-pod content lexica down to procedural synthesisers.

## What changed

A new "procedural lexicon" section in `scripts/pod_eval_vllm.py`
provides four synthesisers that any per-round generator can call:

* `_synthetic_word(rng, n_syll=2)` — pseudo-word from CV(C) syllable
  inventory. Combinatorial space ≈ 5300 syllables → 28M 2-syllable
  words.
* `_synthetic_name(rng)` — Capitalised 2-3-syllable name. Used
  wherever the legacy `_PROC_NAMES` / `_NAMES` lists were referenced.
* `_synthetic_org_topic(rng)` — "the {Adj} {Noun}{suffix} {OrgType}"
  string. Combinatorial space > 1M unique topics. Used by
  `multi_doc_synthesis_bench` instead of the 26-entry
  `_MULTI_DOC_TOPICS` list.
* `_synthetic_entity(rng)` — `(entity_phrase, possessive_phrase,
  item_word)` triple for `long_context_bench` needles. Replaces the
  8-entry `_LC_NEEDLE_TEMPLATES` list.
* `_synthetic_distractor_sentence(rng)` — fully grammatical filler
  sentence ("{Name} {verb}ed the {adj} {noun} {time-modifier}.").
  Replaces the 50-entry `_LC_DISTRACTORS` list.
* `_synthetic_names(rng, n)` — convenience: returns `n` distinct
  procedural names (used by reasoning_bench `web_of_lies` /
  `tracking_objects` / `ordering` / `deduction`).

## Why pseudo-words instead of real words

Real-word vocabularies are bounded — the 50k common-English-word pool
fits in a single training pass. Pseudo-words have unbounded
combinatorial space *and* look natural enough that the model treats
them as named entities (which is what we want — the GRADING never
depends on real-world meaning of these tokens, only on their length
or character composition or position in the document).

A miner who somehow trains specifically on "all 2-syllable
consonant-vowel-coda combinations of the v29.5 syllable inventory"
would have to memorise:

- 37 onsets × 12 nuclei × 12 codas = 5,328 distinct syllables
- 2-syllable words: 28 million combinations
- 3-syllable: 150 billion combinations

Each round per-item samples ~24 syllables. The probability that any
(syllable, role, position) triple from the previous round repeats in
the next is ~zero. There is no useful memorisation surface.

## What's still static (and why)

Some lexicon elements are kept because they carry semantic meaning
the model needs to interpret correctly:

* **Org-type words** (Society / Guild / Festival / Workshop / etc.) —
  the model needs these to recognise "this card describes an
  organisation." 16 fixed entries; mixing pseudo-words here would make
  the question hard to interpret.
* **Roles for needle entities** (archivist / vault keeper / etc.) — same
  reasoning; the model needs these as semantic anchors.
* **Time modifiers** in distractor sentences (before lunch / at the
  morning bell) — same.
* **Verb / adjective / object lexicons** in distractor sentences —
  small fixed pools but composed combinatorially with procedural
  names so each sentence is fresh.

The combinatorial expansion ensures the resulting SENTENCE-LEVEL
surface form rotates per call even though the WORD-LEVEL inventory
has bounded entries. ~8000 names × 23 verbs × 13 adj × 17 noun ×
11 time-mods = 446M unique distractor sentences ⇒ uncacheable.

`debug_bench` and `correction_bench` keep static function names and
templates per kind because:
1. The bug PATTERN (off-by-one / swap-operands / wrong-comparator /
   etc.) is what miners must learn to spot — the function name is
   a distraction.
2. Test-call grading uses `candidate(...)` (not the function name
   directly), so renaming buys nothing.
3. The buggy / corrected function bodies vary by parameters drawn
   from `rng` (`n` for off-by-one ranges, `mod` for modulo bugs,
   `threshold` for comparators, etc.), so the surface form already
   rotates within each kind.

`refactor_bench` has 3 fixed templates × 3 constraint kinds = 9
distinct (template × constraint) instances; the constraint kind
picked per item is procedural, but the templates themselves are
small. Future work will procedurally generate the buggy
implementation from a smaller "kind" taxonomy similar to debug_bench;
deferred to v29.6.

The chat-style probes (`on_policy_rkl`, `judge_probe`,
`chat_turns_probe`) still draw from fixed prompt pools (80 / 65 /
~25 entries), with per-round paraphrase via the v25
`_paraphrase_chat_prompt` helper. Replacing these with fully
procedural prompts is fraught (the teacher-rubric grading needs
coherent input — pseudo-word filler would yield meaningless
grades). Deferred to v29.6 for a careful design pass.

## Forward path

What's left to procedurally synthesise (deferred):

1. **`debug_bench` / `correction_bench` function-name + body
   parameterisation** — currently the (function name, buggy body)
   pair is fixed per bug kind. Combinatorial expansion would require
   procedurally generating valid Python implementations and bug
   injections; substantial work, modest gain (the grading already
   tests the abstract bug pattern).
2. **`refactor_bench` template parameterisation** — same shape
   issue.
3. **Chat-style probe pools (`on_policy_rkl`, `judge_probe`,
   `chat_turns_probe`)** — fully procedural chat prompts risk
   incoherence under teacher-rubric grading. Needs a careful design
   pass (template-based combinatorial synthesis with semantic-slot
   filling).
4. **Org-type / role / verb / noun / adj lexicons inside the lex
   helpers** — kept small because they need to be real English for
   the model to interpret correctly. Could be expanded 5-10× to
   slow down vocabulary memorisation, but combinatorial expansion
   already makes the SENTENCE-LEVEL surface form effectively
   uncacheable.

These are documented in the eval-roadmap.md for future iterations.
