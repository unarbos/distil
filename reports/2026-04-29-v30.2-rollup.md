# SN97 v30.2 — Eval Overhaul: Final Score, Axis Grouping, Super-Teacher, Multi-GPU Prep

**Date:** 2026-04-29
**Schema bump:** composite_shadow_version 28 → 29
**Validator commit:** TBD (this report)

## What shipped

This release implements the recommendations from the v30 strategic
audit (`reports/2026-04-29-v30-strategic-audit.md`):

### 1. New ranking key: `composite.final` (replaces `composite.worst`)

```
final = 0.7 × worst_3_mean + 0.3 × weighted
```

Where `worst_3_mean` is the equal-weighted mean of the 3 lowest
non-broken non-zero-weight axes, and `weighted` is the existing
weighted convex combination.

**Why:** the legacy `worst()` (single-axis min) was dominated by noise
on the lowest-data axis. 22% of scored UIDs (35/160) sat at exactly
`worst = 0` with no way to discriminate. `worst_3_mean` smooths
single-axis variance while preserving anti-Goodhart pressure (~70% of
the score still comes from the bottom of your axis distribution); the
30% weighted contribution stops models being penalised by a single
quirky sub-axis floor.

`worst` and `worst_3_mean` are still emitted as telemetry. Only the
ranking key changes.

### 2. Axis grouping (collapse without losing depth)

Five overlapping bench axes per topic area are now merged into single
group axes for ranking. Each group is the equal-weighted mean of its
sub-axes; the sub-axes still run (bench items still get evaluated)
and still appear in `axes` for telemetry / per_src / saturation
audit. They simply don't directly drive the ranking weight.

| Group axis | Sub-axes | Weight |
| --- | --- | --- |
| `code_skill_group` | code_bench, mbpp_bench, debug_bench, correction_bench, refactor_bench | 0.20 |
| `math_skill_group` | math_bench, aime_bench, robustness_bench | 0.18 |
| `reasoning_skill_group` | reasoning_bench, multi_doc_synthesis_bench, long_context_bench | 0.12 |
| `knowledge_skill_group` | knowledge_bench v2, pragmatic_bench | 0.07 |

Axes that stay separate (different skills): `tool_use_bench`,
`ifeval_bench`, `calibration_bench`, `judge_probe`, `long_form_judge`,
`chat_turns_probe`, `capability`.

**Net cut:** 0.35 weight redirected from per-axis fragmentation to
better signals (super_teacher, RKL bumps, shadow axes promoted).

### 3. Super-teacher axis (incentivize beyond-teacher capability)

Pure distillation cannot exceed the teacher. To produce SOTA-class
small models, miners need to mix in (b) RL on verifiable rewards
(GRPO Stage 4 of the Mining Guide) and (c) post-distillation SFT on
harder data than the teacher saw. The eval now explicitly rewards
both via a new axis:

```
super_teacher = tanh(mean(max(0, student_pass - teacher_pass)) / 0.10)
```

A student that exactly matches the teacher on every verifiable bench
scores 0; a student that beats the teacher by ~0.20 on average scores
~0.95. Default weight 0.10 — significant, but not overwhelming, while
miners learn the new incentive.

The 16 verifiable bench axes contributing: math, code, reasoning,
ifeval, aime, mbpp, tool_use, long_context, robustness, debug,
correction, multi_doc_synthesis, calibration, refactor, pragmatic,
knowledge_bench.

### 4. Shadow axes promoted to research-paper weights

Four shadow axes (entropy_aware_kl, kl_is, forking_rkl,
teacher_trace_plausibility) bumped from 0.02 → 0.05 each (research-
paper recommended weights). Total composite weight injected: 0.20.
The axes have been live for ~3 hours under low weight without
crashing, so promotion is safe.

`final` and `worst_3_mean` are now persisted in `composite_scores.json`

### 7. Multi-GPU pod scaffolding

Env knobs added (`DISTIL_TP_SIZE`, `DISTIL_STUDENT_TP_SIZE`,
`DISTIL_STUDENT_PARALLELISM`) so the operator can migrate the eval
pod from 1× H100 to 4× or 8× H100 without code changes. The
teacher's tensor-parallel size auto-detects from visible GPUs;
student parallelism is a config-only placeholder until the
worker-pool dispatch lands in v30.3.

**Pod size recommendations:**
- Stage 1 (current): 1× H100 80GB. Round wall time ~70 min.
- Stage 2 (recommended next): 4× H100 80GB. Round wall time ~25 min,
  supports v30.2 +50% items.
- Stage 3 (Kimi K2.6 ready): 8× H100 80GB.

### 8. Kimi K2.6 migration prep

Two new modules:

- **`eval/cross_tokenizer.py`** — Path A round-trip helper. Decodes
  Kimi-token-ID generations through the Kimi tokenizer to natural
  text, re-encodes through Qwen tokenizer to feed the existing
  pipeline. Per-position drift score for telemetry. 11 unit tests.

- **`scripts/experiments/run_kimi26_a_b.py`** — A/B experiment
  orchestrator. Runs 5-student × 3-variant × 6-round comparison on a
  separate eval pod. Variants: `qwen36_baseline`, `kimi26_pathA`,
  `kimi26_pathB`. Output goes to `/opt/distil/experiments/...`,
  preserving the production composite_scores untouched.

- **`scripts/experiments/evaluate_kimi26_a_b.py`** — Decision-rule
  evaluator. Applies the tiered gates from the runbook (Tier 1
  disqualifying, Tier 2 promotion, Tier 3 supporting evidence) and
  produces a recommendation: PROMOTE / HOLD / REVERT.

**What's still needed before the swap:**
- vLLM smoke test against K2.6 on the experiment pod (~0.5 day).
- Provision a multi-GPU pod with K2.6 weights (operator).
- Run the 6-round A/B (10 hours pod compute).
- Apply decision rule.

This is unchanged from the runbook; this release just ships the
code that runs the experiment.

## What did NOT change

Per the user's explicit instructions:

- **Winner-takes-all emissions** stay (`build_winner_take_all_weights`
  still puts 1.0 on the king on chain). Top-K linear / softmax
  proposals are documented in the audit but not deployed.
- **One-eval-per-commitment** stays — only the king gets re-evaluated
  each round; non-king commitments are scored once forever (their
  record persists).
- **1-5 rubric grading** stays for `judge_probe` / `long_form_judge`.
  Single-digit emissions are robust to parse and the teacher's
  consistency on a 1-5 scale is empirically tighter than continuous
  scoring would be.
- **6.1 OOD probe**: deferred per user request.
- **6.2 Pairwise judge tournament**: deferred (doesn't fit the 1
  king + N challengers single-eval flow).
- **6.3 Adaptive eval**: deferred (overengineering).
- **6.4 Cross-validator consensus**: skipped (only 1 validator).

## Rollback

Single-flag rollback for any new feature:

- `COMPOSITE_FINAL_BOTTOM_WEIGHT=1.0` → falls back to pure worst_3_mean
  (ignores weighted). Set to 0.0 → pure weighted.
- `CODE_SKILL_GROUP_WEIGHT=0` (etc.) → drops the group axis; manually
  set sub-axis weights back to legacy values to restore v30 behaviour.
- `SUPER_TEACHER_WEIGHT=0` → super-teacher axis dropped from ranking.
- `BENCH_*_PER_ROUND=<old value>` → revert per-axis item counts.

Schema rollback (v29 → v28) is NOT supported — the king filter
quarantines old-schema records, so reverting would leave the validator
kingless until challengers re-evaluate.
